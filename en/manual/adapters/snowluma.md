# SnowLuma Adapter

The SnowLuma Adapter lets MaiBot connect to the QQ platform through [SnowLuma](https://github.com/Mai-with-u/MaiBot-SnowLuma-Adapter), sending and receiving messages and handling group chats and private messages.

::: warning Available (Beta)
The SnowLuma Adapter is currently in beta. Core functionality works, but there may be unhandled edge cases. If you encounter issues, please report them on [GitHub Issues](https://github.com/Mai-with-u/MaiBot-SnowLuma-Adapter/issues).
:::

## Introduction

The SnowLuma Adapter is a MaiBot plugin that communicates bidirectionally with SnowLuma over WebSocket. It works simply:

- **Inbound**: SnowLuma pushes QQ messages to the adapter, which transforms them and injects them into MaiBot
- **Outbound**: Replies generated by MaiBot are transformed by the adapter and sent out through SnowLuma

Unlike the NapCat Adapter, the SnowLuma Adapter only has **plugin mode** — there is no standalone mode. The adapter runs directly inside the MaiBot process, requiring less configuration and simpler deployment.

Message flow: **QQ → SnowLuma → Adapter Plugin (inside MaiBot) → MaiBot**

### Adapter Repository

SnowLuma Adapter source code: [Mai-with-u/MaiBot-SnowLuma-Adapter](https://github.com/Mai-with-u/MaiBot-SnowLuma-Adapter)

## Installation

### Step 1: Get the Adapter

Clone the adapter repository into MaiBot's `plugins/` folder:

```bash
# Enter the MaiBot plugins directory
cd MaiBot/plugins

# Clone the repository
git clone https://github.com/Mai-with-u/MaiBot-SnowLuma-Adapter.git
```

Alternatively, you can install the plugin via MaiBot's WebUI.

### Step 2: Configure the SnowLuma Connection

Since the adapter already runs inside MaiBot, you only need to configure the connection between the adapter and SnowLuma. Edit the adapter config file at `plugins/MaiBot-SnowLuma-Adapter/config.toml` and fill in the SnowLuma address and port.

### Step 3: Configure SnowLuma

Make sure SnowLuma has its forward WebSocket server enabled, and that the listening address and port match the `luma_client.server` and `luma_client.port` settings in the adapter config.

The default connection address is `ws://127.0.0.1:3001`. If SnowLuma runs on the same machine, the defaults will work.

### Step 4: Start

Simply start MaiBot. The adapter will load and connect automatically.

### Plugin is Disabled by Default

After installation, the SnowLuma Adapter plugin is **disabled by default**. You need to enable it manually before it can connect.

#### Method 1: Edit Config File (Recommended)

Edit `plugins/MaiBot-SnowLuma-Adapter/config.toml` and set `enabled` to `true`:

```toml
[plugin]
enabled = true   # Set to true
config_version = "1.0.0"
```

Then restart MaiBot.

#### Method 2: Enable via WebUI

1. Open a browser and go to `http://127.0.0.1:8001`, then log in with your Access Token
2. Click **"Plugin Management"** in the left sidebar
3. Find **"SnowLuma Adapter"** and toggle the enable switch
4. Save the configuration and restart MaiBot (or wait for the plugin hot reload)

> **Verify whether it is enabled**: After starting MaiBot, check the logs for adapter loading and connection success messages. If you see `Disabled in config, skipping activation`, the plugin is not enabled.

## Configuration Reference

The adapter configuration file is located at `plugins/MaiBot-SnowLuma-Adapter/config.toml` and contains the following four sections.

### Plugin Settings (`[plugin]`)

- **`enabled`** — Whether to enable the SnowLuma Adapter. When disabled, the plugin only registers the message gateway and will not actively connect to SnowLuma. Default: false
- **`config_version`** — Current config structure version (auto-managed, generally does not need manual changes). Default: "1.0.0"

### SnowLuma Connection (`[luma_client]`)

- **`server`** — SnowLuma WebSocket server address. Default: "127.0.0.1"
- **`port`** — SnowLuma WebSocket server port. Default: 3001
- **`token`** — SnowLuma access token (can be left empty). Default: empty
- **`connection_id`** — Optional connection identifier, used to distinguish multiple adapter links. Default: empty
- **`reconnect_delay_sec`** — Reconnection delay (seconds) after a connection is lost. Default: 5.0
- **`action_timeout_sec`** — Timeout (seconds) for calling SnowLuma action APIs. Default: 10.0

### Chat Filtering (`[chat]`)

- **`enable_chat_list_filter`** — Whether to enable group and private chat list filtering. When disabled, only `ban_user_id` rules apply. Default: true
- **`show_dropped_chat_list_messages`** — Whether to log messages that were dropped by the chat list filter. Default: false
- **`group_list_type`** — Group chat list mode. Whitelist only accepts messages from listed groups, blacklist ignores messages from listed groups. Default: "whitelist"
- **`group_list`** — List of group IDs in the chat list (automatically deduplicated). Default: empty
- **`private_list_type`** — Private chat list mode. Whitelist only accepts messages from listed users, blacklist ignores messages from listed users. Default: "whitelist"
- **`private_list`** — List of user IDs in the private chat list (automatically deduplicated). Default: empty
- **`ban_user_id`** — Globally blocked user ID list. Messages from these users are dropped before reaching MaiBot. Default: empty
- **`ban_qq_bot`** — Whether to block messages from official QQ bots. Default: false

::: tip Group Whitelist is Enabled by Default
The adapter enables chat list filtering by default, and group chats use whitelist mode by default. Messages from groups not listed in `group_list` will be dropped directly. If the connection succeeds but mentioning the bot in a group gets no response, check this setting first.
:::

During testing, you can temporarily disable list filtering:

```toml
[chat]
enable_chat_list_filter = false
```

Or add the test group to the whitelist:

```toml
[chat]
enable_chat_list_filter = true
group_list_type = "whitelist"
group_list = ["your QQ group number"]
```

### Message Filtering (`[filters]`)

- **`ignore_self_message`** — Whether to ignore messages sent by the bot itself (recommended to keep enabled to prevent the bot from processing its own messages). Default: true

### Full Config Example

```toml
[plugin]
enabled = true
config_version = "1.0.0"

[luma_client]
server = "127.0.0.1"
port = 3001
token = ""
connection_id = ""
reconnect_delay_sec = 5.0
action_timeout_sec = 10.0

[chat]
enable_chat_list_filter = true
show_dropped_chat_list_messages = false
group_list_type = "whitelist"
group_list = ["123456789"]
private_list_type = "whitelist"
private_list = []
ban_user_id = []
ban_qq_bot = false

[filters]
ignore_self_message = true
```

## Verification and Troubleshooting

### Verify Connection

How do you know it is connected? Check these places:

1. **WebUI plugin list**: the SnowLuma Adapter plugin is loaded
2. **MaiBot logs**: shows `SnowLuma WebSocket connected`
3. **Message test**: mention the bot in a QQ group and see whether it replies

### Cannot Connect?

**Check these points**:

- Are the SnowLuma address and port correct? Do they match the adapter's `luma_client.server` / `luma_client.port`?
- Is the firewall blocking the connection?
- Are SnowLuma and MaiBot on the same machine?
- Is the access `token` correct?
- Are there any error messages in the logs?

### Not Receiving Messages?

**Possible causes**:

- The group whitelist is enabled by default — is your group added to `group_list`?
- Is `enabled` set to `true`?
- Did SnowLuma itself receive the message? Check the SnowLuma logs
- Is the network connection normal?

### Cannot Send Messages?

**Troubleshooting**:

- Does the bot have permission to send messages? (Group chats require permissions)
- Check the MaiBot logs for any errors
- Is `action_timeout_sec` set appropriately? You can increase it if network latency is high

### Multi-Instance Deployment

If you need to connect one MaiBot to multiple SnowLuma instances (for example, multiple QQ accounts), you can configure a different `connection_id` for each instance to distinguish the links:

```toml
[luma_client]
connection_id = "bot-1"
```
